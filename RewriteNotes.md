# RuVox 2.0 — План переписывания

Документ фиксирует архитектурные решения для следующей итерации RuVox.

## Мотивация

Текущая реализация (PyQt6 + Python TTS-pipeline) накопила набор болевых точек:

- Долгий старт (torch + PyQt6 + QWebEngineView грузятся eagerly).
- Переключение формата отображения (Markdown ↔ Plain) занимает 1–2 с даже на небольшом тексте — из-за пересборки `QTextDocument`, difflib-mapping'а и повторного rendering'а.
- Эпизодические подлаги во время воспроизведения (GIL + O(N) поиск по timestamps + синхронная работа с QTextDocument на каждом тике плеера).
- `QWebEngineView` (весь Chromium) подтягивается только ради mermaid-диаграмм.
- Препроцессинг текста по времени сопоставим с самим TTS: 25+ последовательных regex-проходов, ручной `CharMapping`.

Отдельно — желание реализовать фичи из roadmap'а, для которых текущий стек плохо подходит:
- FF 1.1 Preview-диалог нормализации.
- FF 1.2 Редактирование текста перед TTS.
- Поддержка HTML-формата с сохранением оформления.

## Стек

| Компонент | Выбор |
|-----------|-------|
| Shell / UI | Tauri 2 |
| Frontend | React + TypeScript + Mantine (версия 8+) |
| Стилизация | Дефолтная тема Mantine, CSS-переменные, без кастомного CSS на старте |
| TTS Pipeline | Rust (`regex` + `aho-corasick`) |
| Storage | JSON (`serde_json`), схема-совместимая с текущей Python-версией |
| Аудио-плеер | `libmpv-rs` |
| TTS-движок | Silero через Python-subprocess |
| Дистрибуция | Nix flake |

## Решения и обоснования

### 1. Tauri + React + Mantine (не Solid/Svelte, не PyQt)

**Почему Tauri.** Webview нативный → старт <500 мс, HTML/Markdown/Mermaid/подсветка кода — в DOM нативно, preview/edit (FF 1.1, 1.2) реализуются как стандартные веб-компоненты. Размер бинарника 10–30 МБ.

**Почему React.** Активное использование LLM для UI-кода. У React экосистема и обучающая выборка на порядок больше, чем у Solid/Svelte — Claude/Copilot генерируют корректный код без доп. инструкций.

**Почему Mantine.** Готовая component-библиотека закрывает ~80% нужных виджетов (Button, Modal, Drawer, Notifications, Form, Table, SegmentedControl и т.д.) плюс `@mantine/hooks` (80+ утилит). Pet-проект — ценность «взял готовое» выше, чем «вылизал каждый пиксель» (что даёт Tailwind+shadcn).

**Caveat по Mantine и LLM.** Был breaking change 6 → 7 (переход с emotion на CSS Modules + CSS-переменные). LLM-ы смешивают v6 и v7 синтаксис. **В CLAUDE.md явно указать:** «используем Mantine 8, стилизация через CSS Modules и prop `classNames`, никакого `sx`/`createStyles`, форма через `@mantine/form` (не react-hook-form)».

### 2. Дефолтная тема Mantine + переключатель

Никакого кастомного CSS на старте. `MantineProvider` с `defaultColorScheme="auto"` (следует системной `prefers-color-scheme`). `primaryColor` — дефолтный.

**Переключатель темы включён в базовый набор.** `SegmentedControl` с тремя опциями: `Light` / `Dark` / `Auto`. Размещение — маленькая иконка/контрол в header'е главного окна (не в настройках — быстрее доступ).

Персистенция — через стандартный `localStorageColorSchemeManager` Mantine. В будущем (polish) можно прокинуть кастомный `colorSchemeManager` с записью в единый `config.json` через Tauri IPC.

Старые PyQt-темы (`Steel Terminal` и др. из `themes.py`) — выбросить при миграции.

### 3. Pipeline на Rust

Одна из главных выгод переписывания: препроцессинг сейчас занимает столько же, сколько сам TTS.

- `regex` крейт (DFA) — 10–100× быстрее Python `re`.
- `aho-corasick` для словарных lookup'ов (IT_TERMS, AS_WORD, GREEK_LETTERS).
- Один проход лексера: текст → массив типизированных спанов (`Word`, `Url`, `Code`, `Number`, …) → диспетчер нормализаторов на спан. Вместо 25 последовательных regex-проходов.
- `CharMapping` строится встроенно, без dual-pass.

**Golden-tests:** сохранить Python-реализацию как reference, для каждого `(input, expected_output)` Rust должен выдавать то же. Снапшот-тесты — и регрессия, и живая документация.

### 4. Python только для Silero, через subprocess

**Не PyO3** (не эмбеддим CPython в Tauri-процесс). Причины:
- Исторический SIGSEGV в torch BERT при неаккуратной инициализации → любой краш Python утащит UI.
- GIL прямо в Tauri-процессе — лишний источник багов.
- Сборка Tauri + embedded Python на практике — боль.

**Не systemd-daemon.** Subprocess под lifecycle Tauri: спавнится при старте, убивается при выходе. Проще и настройки, и дебага. Вопрос долгоживущего daemon'а (с персистентной моделью между сессиями) — отложен до тех пор, пока ждать прогрев при старте реально начнёт раздражать.

**Протокол — минимальный.** Не стримим аудио чанками, не заморачиваемся с framing бинарных данных:

```
stdin  (одна строка JSON):
  {"text": "...", "speaker": "v5_ru", "sample_rate": 48000, "out_wav": "/tmp/ruvox/xyz.wav"}

stdout (одна строка JSON):
  {"ok": true, "timestamps": [...], "duration_sec": 12.3}
  {"ok": false, "error": "..."}
```

- Rust генерирует уникальный `out_wav` path, после успеха забирает WAV с диска.
- Один in-flight request за раз (Silero не thread-safe) → очередь живёт в Rust, корреляция-id не нужна.
- Дебаг прост: WAV можно открыть руками.
- Для новых параметров — опциональные поля в JSON.
- Логи Python на stderr → Rust проксирует в `tracing`.

**Прогрев модели.** Tauri спавнит subprocess в `on_setup`, шлёт первым сообщением `warmup`-команду. К моменту, когда пользователь копирует первый текст, модель загружена.

### 5. Storage — JSON as-is

Схема текущего `history.json` и сопутствующих файлов сохраняется:
- `~/.cache/ruvox/history.json` — версионированный список `TextEntry`.
- `~/.cache/ruvox/audio/{uuid}.wav` — аудио.
- `~/.cache/ruvox/audio/{uuid}.timestamps.json` — тайминги слов.

**Бонус:** схема-совместимость → при переходе не нужна миграция кэша, Rust-версия читает существующий `history.json`. Старое PyQt-приложение и новое Tauri-приложение могут работать с одним кэшем (не одновременно).

На Rust-стороне — `serde_json` + `Vec<TextEntry>` в памяти. SQLite — когда реально понадобится (тысячи записей), не раньше.

### 6. libmpv через `libmpv-rs`

Сохраняем текущий плеер как есть — там есть качественное масштабирование скорости (scaletempo2), критичное для прослушивания на > 1.0x. `libmpv-rs` — биндинги к той же libmpv, что используется сейчас через `python-mpv`. Минимум изменений поведения.

### 7. Хоткеи — не первоочередная фича

По опыту использования: хоткеи редко критичны — обычно пользователь всё равно смотрит в окно при чтении. Реализация хоткеев (через `tauri-plugin-global-shortcut`) — опциональная фича после MVP, не блокер.

## Отложенные вопросы

### Silero и зависимость от torch

Silero заброшен автором (нет активной разработки). Надежда на community ONNX-конверсию слабая. Pet-проект **живёт с Python-subprocess + torch до тех пор, пока:**
- Silero сам по себе удовлетворяет по качеству русского голоса (пока альтернатив с тем же соотношением качество/скорость/оффлайн нет).
- Размер окружения и время прогрева остаются терпимыми.

**Возможные будущие направления** (не приоритет):
- Самостоятельный экспорт Silero в ONNX (сложно из-за TorchScript + BERT homosolver).
- Миграция на другой движок (Piper, VITS-based fine-tunes, XTTS-v2).
- Периодически (раз в квартал) проверять HuggingFace на свежие русские TTS-модели.

Когда/если torch-зависимость уйдёт — Python-subprocess заменяется на Rust-крейт (`ort` — биндинги к onnxruntime).

### Персистенция выбора темы в едином config'е

Сейчас — `localStorage`. Полноценный `colorSchemeManager`, пишущий в общий `config.json` через Tauri IPC — задача на polish-фазу.

### Кэш нормализации

Не описан выше явно, но закладывается дизайном: при pipeline на Rust все результаты (`normalized_text`, `char_mapping`, `timestamps`) хранятся в `TextEntry`. Повторное прослушивание, preview, edit — из кэша без пересчёта.

## План фаз

| Фаза | Что | Срок (вечерами) |
|------|-----|-----------------|
| 0 | **Спайк.** Пустой Tauri + Python-subprocess, читающий stdin и генерирующий WAV. Хоткей → clipboard → subprocess → WAV → `libmpv-rs`. Проверка: субпроцесс-sidecar работает через Nix, `libmpv-rs` играет со scaletempo2, DX React+Mantine комфортный. | 1 нед |
| 1 | Перенос pipeline на Rust. Golden-tests из Python-версии. Без UI-фишек. | 2–3 нед |
| 2 | UI: очередь + viewer (Markdown + plain) + плеер + подсветка слов + переключатель темы. | 2–3 нед |
| 3 | Mermaid (нативно в DOM) + HTML-режим + настройки. | 2 нед |
| 4 | Preview/edit (FF 1.1, 1.2). | 1 нед |
| 5 | Дистрибуция через Nix flake, полировка, тесты. | 1–2 нед |

Ориентировочно 9–12 недель по вечерам без давления.

## Контрольные точки (exit criteria Фазы 0)

Продолжать на Фазу 1 имеет смысл, если:
1. Python-subprocess стартует и общается с Rust стабильно под Nix flake.
2. `libmpv-rs` играет с scaletempo2 (качество на 1.5x–2.0x — как в текущей версии).
3. Время старта Tauri-приложения до готового к вводу UI — <1 с.
4. DX с React + TypeScript + Mantine субъективно комфортен.

Если хотя бы один пункт «нет» — дёшево возвращаемся к альтернативам: G2 (Python-стек + subprocess TTS + кэш + markdown-it-py) или G3 (Rust pipeline через PyO3 при сохранении PyQt UI).

## Рабочий процесс: git worktree + параллельная миграция

Разработка новой версии идёт на **отдельной ветке в отдельном worktree** — текущий PyQt-RuVox остаётся рабочим на `main` всё время переписывания.

### Настройка

```bash
# из корня репозитория на main
git worktree add ../RuVox-tauri -b rewrite/tauri

# дальше работаем в:
cd ../RuVox-tauri
```

- Имя ветки: `rewrite/tauri` (можно любое другое, например `v2` или `tauri`).
- Путь worktree: соседняя папка `../RuVox-tauri/` (не вложенная в основной репозиторий).

### Что делится, а что раздельно

| Ресурс | Поведение |
|--------|-----------|
| Git history | Общий (один репозиторий, разные ветки). |
| Файлы в рабочем дереве | Раздельные (worktree — это физическая копия checkout'а). |
| Сборочные артефакты (`target/`, `node_modules/`, `.venv/`) | Раздельные — каждому worktree своё. |
| `shell.nix` / `flake.nix` | В rewrite-ветке свой (rust toolchain + node + python-для-ttsd), в main-ветке текущий. |
| Кэш приложения (`~/.cache/ruvox/`) | **Общий**. Схема `history.json` совместима → можно переключаться между версиями приложения, история и аудио сохраняются. |
| Логи и debug-файлы | Общие по умолчанию — но можно развести через env-переменную в новой версии, если начнут мешать. |

### Плюсы подхода

- **IDE открыт на обоих worktree одновременно** — сравнение референса (старой реализации) с новой идёт буквально в соседних окнах.
- **Можно быстро переключиться на main** для критичного багфикса PyQt-версии, не теряя состояние rewrite (никаких `git stash`).
- **Golden-tests** новой Rust pipeline запускаются против фиксированных expected'ов, сгенерированных из Python на main — без смены веток.
- **Сравнение поведения бок-о-бок:** одну и ту же статью кладём в clipboard → запускаем обе версии (PyQt-RuVox из main, Tauri-RuVox из rewrite) на разных тестовых аудио-устройствах / в разное время. Кэш общий → переозвучка не нужна.

### Структура файлов в rewrite-ветке

Tauri использует стандартный layout:
- `src/` — React+TypeScript фронт.
- `src-tauri/` — Rust-код (pipeline, storage, libmpv, subprocess-менеджер).
- `ttsd/` (рабочее имя) — Python-subprocess с Silero.

Текущее Python-приложение (`src/ruvox/`) при старте rewrite-ветки **переносится в `legacy/ruvox/`** — это избегает коллизии с Tauri-овским `src/` и оставляет референс для копирования логики. Когда rewrite достигнет фичи-паритета — `legacy/` удаляется одним коммитом.

### Финализация

Когда rewrite достигнет фичи-паритета и пройдёт приёмку:
1. Merge `rewrite/tauri` → `main` (обычный merge commit, без squash — история переписывания полезна для будущих археологов).
2. Удаление `legacy/` отдельным follow-up коммитом.
3. `git worktree remove ../RuVox-tauri` — worktree больше не нужен.
4. Ветка `rewrite/tauri` может остаться в истории или быть удалена — на усмотрение.
